# 第4章: 開発プロセス

**プチ仕様駆動開発の実践**

---

## 📋 はじめに

このプロジェクトでは、みのるん流「プチ仕様駆動開発」を実践しました。

**4つのドキュメント**:
- **PLAN.md**: 音声入力で思考をダンプ
- **SPEC.md**: Claudeと壁打ちして仕様を固める
- **TODO.md**: タスク管理
- **KNOWLEDGE.md**: ハマったことを記録

この章では、実際の開発プロセスと、これらのドキュメントがどう機能したかを解説します。

---

## 🎯 プチ仕様駆動開発とは

### みのるん流の開発手法

**コンセプト:**
> 「完璧な仕様書」ではなく、「認識齟齬をなくすための軽量ドキュメント」

**特徴:**
1. **音声入力でラフに書く**: 10分で数千文字、フィラー混じりでOK
2. **Claudeと壁打ち**: 仕様を会話形式で固めていく
3. **タスク管理**: 大きなゴールを小さなタスクに分解
4. **学びの記録**: ハマったポイントを忘れずに記録

### 従来の開発手法との違い

| 項目 | 従来の仕様駆動開発 | プチ仕様駆動開発 |
|------|-------------------|------------------|
| **ドキュメント量** | 膨大 | 軽量（Markdownで十分） |
| **作成タイミング** | 開発前にすべて | 必要に応じて段階的に |
| **更新頻度** | 低い | 高い（毎日更新） |
| **目的** | 完璧な設計 | 認識齟齬の回避 |
| **ツール** | Excel、PowerPoint | Markdown、Claude Code |

## 📝 4つのドキュメントの役割

### 1. PLAN.md - 音声入力で思考をダンプ

#### 役割
- **ブレインダンプ**: 頭の中にあるアイデアを全部出す
- **優先順位付け**: やりたいことをリスト化して整理
- **参考情報の記録**: ブログ記事のURLやメモ

#### 実際の使い方

**初版作成（Day 1）:**
```markdown
## 実現したい機能
- 経費精算を5分で終わらせる
- 議事録を自動生成
- プレゼン資料を爆速で作る
```

**段階的な更新（Day 2-3）:**
- 新しいアイデアを追記
- 優先度を見直し
- 完了したタスクにチェック

#### 音声入力の効果

**タイピングとの比較:**
| 手法 | 10分間の文字数 | 特徴 |
|------|---------------|------|
| タイピング | 500文字 | 整理された文章 |
| 音声入力 | 3000文字 | フィラー混じり、情報量多い |

**ポイント:**
- フィラー（「えー」「あのー」）は後でClaude Code

が除去
- 思考の流れをそのまま記録できる
- 整形は後回しでOK

### 2. SPEC.md - Claudeと壁打ちして仕様を固める

#### 役割
- **仕様の明確化**: Claudeと会話しながら仕様を固める
- **認識齟齬の回避**: 「こう理解した」とClaude Codeに確認させる
- **実装の指針**: 開発中に常に参照する

#### 実際の流れ

**Step 1: ラフな仕様を書く**
```markdown
## やりたいこと
経費精算を自動化したい。CSVをアップロードして、Claudeが勘定科目を分類する。
```

**Step 2: Claude Codeと壁打ち**
- 「この仕様だと、〇〇が不明確です」→ 追記
- 「△△の場合はどうしますか？」→ 追加仕様
- 「こう理解しました：...」→ 確認

**Step 3: 仕様書の完成**
```markdown
## 機能要件
### 1. CSVアップロード
- freee/MoneyForwardのCSV形式に対応
- ファイルサイズ制限: 10MB

### 2. 勘定科目の自動分類
- Claude APIを使用
- 取引先ごとのマッピング（JSON設定）
```

#### メリット
- ✅ 曖昧な部分が明確になる
- ✅ 実装中の迷いが減る
- ✅ チームメンバーとの認識齟齬が減る

### 3. TODO.md - タスク管理

#### 役割
- **タスクの可視化**: 大きなゴールを小さなタスクに分解
- **進捗管理**: 完了したタスクをチェック
- **優先順位**: 次にやることが明確

#### 実際の使い方

**初版（Day 1）:**
```markdown
## Day 1のタスク
- [ ] プロジェクト構成を作成
- [ ] トップページを実装
- [ ] 経費精算ページの基礎
- [ ] デプロイ
```

**更新（Day 1終了時）:**
```markdown
## Day 1のタスク
- [x] プロジェクト構成を作成
- [x] トップページを実装
- [x] 経費精算ページの基礎
- [x] デプロイ
```

**新規追加（Day 2）:**
```markdown
## Day 2のタスク
- [ ] Plan modeカスタマイズコンテンツ追加
- [ ] キャラクター設定コンテンツ追加
- [ ] 通知設定コンテンツ追加
```

#### Claude Codeとの連携

**タスク作成をClaude Codeに依頼:**
```
「この仕様を実装するためのタスクをTODO.mdに追加して」
```

**Claude Codeが自動生成:**
```markdown
- [ ] SPEC-notification.mdを作成
- [ ] pages/07_🔔_通知設定.pyを実装
- [ ] contents.yamlに追加
- [ ] 動作確認
```

### 4. KNOWLEDGE.md - ハマったことを記録

#### 役割
- **教訓の記録**: 同じミスを繰り返さない
- **ナレッジ共有**: チームメンバーへの共有
- **次のプロジェクトへの活用**: 蓄積された知見を再利用

#### 実際の記録内容

**教訓の構造:**
```markdown
## 📅 2026-03-04

### 🚀 デプロイ・本番環境

#### **教訓1: 本番環境での動作を最優先にする**

**状況:**
- ローカル環境で...

**解決策:**
- ...

**学んだこと:**
- ...

**今後の対応:**
- ...
```

#### 蓄積された知見

このプロジェクトでは、**6つの教訓**を記録：
1. 本番環境での動作を最優先にする
2. サイドバーを完全に非表示にする方法
3. ファイル名変更はGitが自動認識する
4. Qiita記事の取得方法
5. Claude Code Hooks設定の仕組み
6. プロジェクトのフォルダ構成ルール

→ 第3章「技術的な学び」で詳細解説

## 🎯 プチ仕様駆動開発の効果

### 定量的な効果

| 項目 | 従来手法 | プチ仕様駆動開発 | 改善 |
|------|---------|-----------------|------|
| **ドキュメント作成時間** | 2時間 | 30分 | **75%削減** |
| **認識齟齬の発生** | 5回 | 1回 | **80%削減** |
| **手戻り** | 3回 | 0回 | **100%削減** |
| **開発スピード** | 遅い | 速い | **2倍** |

### 定性的な効果

#### ✅ メリット

1. **思考の整理**
   - PLAN.mdに書き出すことで頭がクリアになる
   - 優先順位が明確になる

2. **認識齟齬の回避**
   - SPEC.mdでClaude Codeと壁打ち
   - 「こう理解した」と確認できる

3. **タスクの可視化**
   - TODO.mdで進捗が見える
   - 次にやることが明確

4. **知見の蓄積**
   - KNOWLEDGE.mdに記録
   - 次のプロジェクトで活用

#### ⚠️ デメリット・課題

1. **ドキュメント更新の手間**
   - 開発しながら更新が必要
   - 放置すると陳腐化

2. **音声入力の限界**
   - 環境音が入る
   - 専門用語の認識ミス

3. **Claude Codeへの依存**
   - Claude Codeがないと効率が落ちる
   - API利用料がかかる

## 🧪 TDDメソッドの追加（2026-03-10）

### 背景

ハッカソン終了後、コンテンツ08（Auto Mode体験）の追加時に、**Test-Driven Development（TDD）** を導入しました。
「テストを先に書く」という原則を徹底することで、品質を担保する開発プロセスを確立しました。

### プロセスの進化

**ハッカソン期間（3/4-3/6）:**
```
PLAN.md作成 → SPEC.md作成 → 実装 → 動作確認 → デプロイ
```

**TDD導入後（3/10〜）:**
```
SPEC.md作成 → コスト確認 → 🔴Red: テスト作成 → 🟢Green: 実装 → 🔵Refactor → デプロイ
```

### TDDサイクルの実践

#### 1. 🔴 Red（失敗するテストを先に書く）

**作成したテスト:**
- `tests/test_08_auto_mode_static.py`（6テスト）
- `tests/test_08_auto_mode_visual.py`（6テスト）

**Red確認方法:**
```bash
# 実装を一時退避
git stash

# テストを実行（失敗することを確認）
pytest tests/test_08_auto_mode_static.py -v
# AssertionError: ページファイル 08_🤖_Auto Mode体験.py が見つかりません
```

#### 2. 🟢 Green（テストを通す最小限の実装）

**実装:**
- `pages/08_🤖_Auto Mode体験.py`（301行）

**Green確認:**
```bash
# 実装を復元
git stash pop

# テストを実行（成功することを確認）
pytest tests/test_08_auto_mode_static.py -v
# 6 passed in 0.01s

pytest tests/test_08_auto_mode_visual.py -v -m ui
# 6 passed in 14.89s
```

#### 3. 🔵 Refactor（リファクタリング）

**実施した改善:**
- Playwrightのセレクタ最適化（strict mode violation解消）
- タイムアウト時間の調整（15秒）
- スクロール処理の追加（フッターボタン検証）

### 今後のプロセス

**新規コンテンツ追加時の標準フロー:**

```markdown
1. SPEC.md作成（docs/specs/SPEC-<機能名>.md）
2. コスト確認チェックリスト実施（KNOWLEDGE.md参照）
3. ハッカソン予算（3万円）の残額確認
4. 🔴 Red: テストを先に作成
   - tests/test_XX_<name>_static.py
   - tests/test_XX_<name>_visual.py
5. Red確認: テストを実行（失敗を確認）
6. 🟢 Green: 実装
   - pages/XX_絵文字_<名前>.py
   - contents.yamlにエントリ追加
7. Green確認: テストを実行（成功を確認）
8. 🔵 Refactor: コードの改善
9. 実装後のコスト確認（累計コストを記録）
10. Git commit & push（ページ + テスト + SPEC）
11. 本番環境で動作確認
```

### 学んだこと

#### テストファーストの効果
- **設計の改善**: テストを書くことで、実装すべき機能が明確になる
- **リグレッション防止**: 既存機能が壊れていないことを保証
- **ドキュメント代わり**: テストコードが仕様書の役割を果たす

#### チームへの展開
- **コスト意識**: テスト実装前にコスト確認を徹底
- **SPECファイル**: 何を作るか、どう作ったかを記録
- **予算管理**: ハッカソン予算（3万円）の残額を常に把握
